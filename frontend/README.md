# Orbit — frontend

React + TypeScript + Vite client for the Orbit RPO simulation platform.
Renders the dual-viewport UI (Cesium global view + three.js proximity view)
that the backend's propagation engine streams state into.

The big picture is in [../docs/architecture-and-roadmap.md](../docs/architecture-and-roadmap.md).
This README is the frontend-specific operating manual.

## Recommended: run via Docker Compose from the project root

```sh
cd ..
docker compose up -d --build
```

Then open **http://localhost:5174/** (or your server's address on port 5174;
the docker-compose mapping is `host:5174 → container:5173`).

The Vite dev server proxies `/api/*` to the backend service over the docker
network, so the browser only ever talks to port 5174. No CORS in dev.

## Local frontend dev (against an externally-running backend)

For tighter iteration on the frontend alone:

1. Install deps on the host:

   ```sh
   npm install
   ```

2. Set up env vars:

   ```sh
   cp .env.example .env
   # then fill in VITE_CESIUM_ION_TOKEN from ion.cesium.com/signup
   # and adjust PROXY_TARGET if your backend isn't at http://localhost:8081
   ```

3. Make sure a backend is running and reachable at `PROXY_TARGET` (the simplest
   way is `docker compose up -d backend db` from the project root, which
   publishes the backend on `localhost:8081`).

4. Start Vite:

   ```sh
   npm run dev
   ```

   App at **http://localhost:5173/**.

## Regenerating the OpenAPI client

The typed REST client in `src/api/` is generated from the backend's running
OpenAPI spec. Re-run after any backend contract change:

```sh
npm run gen:api
```

This requires a backend reachable at `$PROXY_TARGET` (defaults to
`http://localhost:8081`).

## Production build

```sh
npm run build       # static output in dist/
npm run preview     # smoke-test locally
```

The reverse proxy in front of the static `dist/` serves the same `/api`
prefix to the backend.

## Working with the dev container — the dep-change workflow

The dev `docker-compose.yml` uses two volume mounts on the frontend service:

```yaml
volumes:
  - ./frontend:/app          # bind: host source → container
  - /app/node_modules        # anon volume: container's node_modules
```

This pairing **looks** redundant but is doing a specific job:

- The bind mount surfaces your host edits inside the container so Vite HMR
  fires on save. Without it, edits don't reach the container.
- The anonymous volume on `/app/node_modules` **carves out** that subdirectory
  from the bind mount so the container reads its own (image-installed) Linux
  `node_modules`, not whatever happens to be in your host's `node_modules`.

When the container is created for the first time, Docker copies the image's
`/app/node_modules` *into* the anon volume so it has the deps the image was
built with.

### The gotcha

**Anonymous volumes persist across container recreations.** Rebuilding the
image with new deps does not refresh the volume — Docker reuses the existing
one. Result: image has the new deps; container reads the old volume; you see
"Failed to resolve import 'X'" in the browser.

### The reliable workflow when frontend deps change

Whenever you edit `package.json` (added/removed/bumped a dep, or even just
`@types/...`):

```sh
# from project root
docker compose build frontend            # rebuild image with new node_modules
docker compose rm -v -s -f frontend      # -v deletes the container's anon volumes too
docker compose up -d frontend            # next start re-populates the anon vol from the image
```

`-v` on `rm` (or `down -v`, which also wipes db_data — usually not what you
want) is the key flag. Without it the anon volume sticks around and the
rebuild is silently undone.

Pure-source edits (no `package.json` change) need none of this — HMR handles
them through the bind mount automatically.

## Data sources

- TLE data: [CelesTrak](https://celestrak.org/) — pulled by the **backend's**
  catalog service in Phase 2+, not the frontend.
- Earth imagery: [Cesium ion](https://ion.cesium.com/) — free up to 5 GB
  egress per month.
